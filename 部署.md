# SceneTAP 部署说明

## 1. 这两个仓库分别是干什么的

### SoM

仓库地址：<https://github.com/microsoft/SoM>

`SoM` 全称是 `Set-of-Mark`。它的核心思想是先对图像做分割，再把每个候选区域用数字或字母标出来，从而增强大模型对图像局部区域的指代和定位能力。

在 `SceneTAP` 里，`SoM` 不是用来生成对抗文字本身的，而是用来做前处理：

- 对原图做区域分割。
- 生成带编号的分割可视化图。
- 导出每个区域对应的 mask。
- 为后续攻击规划器提供“把文字放在哪个区域”这一空间基础。

对应到当前仓库，`SoM` 主要被下面这条链路使用：

- `save_som_images.py`
- `utils/som.py`
- `SoM/` 目录下的配置、模型和算子

也就是说，`SoM` 在 `SceneTAP` 里的角色可以理解为：

`场景区域发现器 + 可编号的空间提示生成器`

### TextDiffuser-2

仓库地址：<https://github.com/microsoft/unilm/tree/master/textdiffuser-2>

`TextDiffuser-2` 是一个面向图像文字生成/编辑的模型，强调：

- 根据文本提示规划文字布局。
- 在图像中生成具有指定内容的文字。
- 支持 text rendering、template-based generation、text inpainting 等任务。

在 `SceneTAP` 里，它的作用不是做语义分割，而是把攻击文本“自然地写进图像里”。也就是：

- 根据攻击规划器选中的目标区域。
- 在该区域内进行文字扩散生成或 inpainting。
- 让最终对抗文字更像场景中的自然文字，而不是生硬贴图。

对应到当前仓库，`TextDiffuser-2` 主要体现在：

- `utils/text_diffuser.py`
- `utils/typo_attack_planner.py`

可以把它理解为：

`对抗文字渲染器 / 区域文字编辑器`

### 两者在 SceneTAP 中的分工

`SceneTAP` 的整体流程可以简化成两步：

1. `SoM` 先回答“哪里适合放文字”。
2. `TextDiffuser-2` 再回答“怎样把文字自然地放进去”。

因此，这两个仓库并不是功能重复，而是上下游关系：

- `SoM` 负责空间定位。
- `TextDiffuser-2` 负责视觉生成。

## 2. 当前仓库实际是怎么用它们的

当前仓库的 `README.md` 只放了两个上游链接，但结合代码可以看到更具体的依赖关系。

### 阶段一：生成 SoM 分割结果

`save_som_images.py` 会：

- 读取原始图片和问题文件。
- 加载 `SoM` 的三个权重：
  - `SoM/swinl_only_sam_many2many.pth`
  - `SoM/sam_vit_h_4b8939.pth`
  - `SoM/seem_focall_v1.pt`
- 调用 `utils/som.py` 进行分割。
- 输出：
  - 带编号的分割图
  - 每个区域的 mask（`.npy`）

这些结果后面会被攻击规划器读取。

### 阶段二：生成对抗文字图像

`chatgpt_test.py` 会：

- 读取第一阶段产出的 SoM 分割图和 mask。
- 调用 `TypoAttackPlanner` 让大模型规划：
  - 要攻击什么
  - 错误诱导答案是什么
  - 文字放在哪个分割区域
  - 用什么短文本植入场景
- 再调用 `TextDiffuser` 把对抗文字渲染到目标区域中。

### 一个很关键的结论

对当前 `SceneTAP` 代码路径来说，推荐把 `SoM` 和 `TextDiffuser-2` 拆成两个环境来跑。

原因是：

- `save_som_images.py` 只依赖 `SoM`。
- `chatgpt_test.py` 依赖 `TextDiffuser` 和 OpenAI API。
- 两个脚本是串行执行，不要求必须在同一个 Python 环境中完成。
- `SoM` 和 `TextDiffuser-2` 的官方依赖版本并不完全一致。

因此最稳妥的方案是：

- 一个 `som` 环境跑分割。
- 一个 `textdiffuser2` 环境跑攻击生成。

## 3. 推荐部署方式

## 3.1 操作系统建议

强烈建议在 `Linux` 或 `WSL2 Ubuntu` 中部署，而不是原生 Windows。

原因：

- `SoM/ops/make.sh` 是 Bash 脚本。
- `SoM` 里有 CUDA 扩展编译。
- 上游仓库默认按 Linux 路径和工具链编写。
- `TextDiffuser-2` 也默认按 CUDA + Linux 环境给出依赖组合。

如果你当前主机是 Windows，推荐方案是：

- 在 `WSL2 + Ubuntu 22.04` 下部署。
- 或直接在一台 Linux 服务器上部署。

## 3.2 GPU 与基础依赖

建议至少准备：

- NVIDIA GPU
- 正常可用的 CUDA
- `conda` 或 `miniconda`
- `git`

额外建议：

- 先确认 `nvidia-smi` 可用。
- 再确认 Python 环境中 `torch.cuda.is_available()` 为 `True`。

## 4. 部署总览

推荐流程：

1. 克隆当前仓库。
2. 建立 `som` 环境并完成 `SoM` 依赖、权重和算子编译。
3. 运行 `save_som_images.py` 生成分割结果。
4. 建立 `textdiffuser2` 环境并安装 `TextDiffuser-2` 相关依赖。
5. 配置 `OPENAI_API_KEY`。
6. 运行 `chatgpt_test.py` 完成 SceneTAP 攻击。

## 5. 第一步：准备当前仓库

当前仓库里已经自带了 `SoM/` 目录，因此不一定需要额外单独克隆 `microsoft/SoM`。

```bash
git clone <your-scenetap-repo>
cd scenetap
```

后续命令默认都在仓库根目录执行。

## 6. 第二步：配置 SoM 环境

## 6.1 创建环境

上游 `SoM` 的部署脚本里出现了 `python3.10`，因此建议这里直接使用 `Python 3.10`。

```bash
conda create -n scenetap-som python=3.10 -y
conda activate scenetap-som
```

## 6.2 安装 SoM 基础依赖

根据上游 README，`SoM` 的核心依赖包括：

- `SEEM`
- `segment-anything`
- `Semantic-SAM`
- `detectron2`
- 自定义 CUDA deformable attention 算子

建议按上游 README 的顺序安装：

```bash
# 假设你已将仓库克隆到本地（注意必须切换到 package 分支）
# 比如目录名为 Segment-Everything-Everywhere-All-At-Once
pip install -e ./Segment-Everything-Everywhere-All-At-Once

pip install git+https://github.com/facebookresearch/segment-anything.git
pip install git+https://github.com/UX-Decoder/Semantic-SAM.git@package
python -m pip install "git+https://github.com/MaureenZOU/detectron2-xyz.git"
```

## 6.3 编译 SoM 的 CUDA 扩展

`SoM` 依赖 `ops/` 下的自定义算子。上游给出的编译方式是：

```bash
cd SoM/ops
bash make.sh
cd ../..
```

注意：

- 这一步需要 CUDA 工具链可用。
- `make.sh` 会尝试读取 `nvcc` 版本。
- 如果 `CUDA_HOME` 未设置，编译可能失败。

如果提示 `CUDA_HOME is None`，先设置：

```bash
export CUDA_HOME=/usr/local/cuda
```

## 6.4 下载 SoM 权重

当前仓库的 `save_som_images.py` 固定读取以下文件：

- `SoM/swinl_only_sam_many2many.pth`
- `SoM/sam_vit_h_4b8939.pth`
- `SoM/seem_focall_v1.pt`

可以直接使用仓库里现成的下载脚本：

```bash
cd SoM
bash download_ckpt.sh
cd ..
```

下载脚本本质上会拉取：

- `Semantic-SAM` checkpoint
- `SEEM` checkpoint
- `SAM ViT-H` checkpoint

## 6.5 安装当前仓库第一阶段还需要的通用包

为了运行 `save_som_images.py`，建议补齐这些通用依赖：

```bash
pip install numpy pillow scipy opencv-python pytorch-lightning
```

## 6.6 验证 SoM 环境

可以先做两个最小检查。

检查 1：CUDA 是否可用

```bash
python -c "import torch; print(torch.cuda.is_available())"
```

检查 2：是否能导入 SoM 封装

```bash
python -c "from utils.som import SoM; print('SoM import ok')"
```

如果这里能通过，说明第一阶段基本可用。

## 7. 第三步：运行 SoM 阶段

使用 `scenetap-som` 环境运行：

```bash
python save_som_images.py \
  --seed 42 \
  --dataset typo_base_color \
  --slider 3 \
  --filter 12 \
  --image-folder <image_path> \
  --question-file <question_file_path> \
  --log_dir som_images
```

执行完成后，典型输出目录类似：

```text
som_images/<dataset>/slider_<value>/seed_<value>/filter_<value>/
```

里面会包含：

- 分割可视化图片
- 同名 `.npy` mask 文件

这些内容会在第二阶段被直接读取。

## 8. 第四步：配置 TextDiffuser-2 环境

## 8.1 为什么建议单独建环境

`TextDiffuser-2` 官方 README 推荐 `Python 3.8`，并且依赖：

- `diffusers==0.24.0`
- `accelerate==0.22.0`
- `datasets==2.11.0`
- `transformers==4.28.1`
- `fschat==0.2.26`

同时还需要与你 CUDA 匹配的：

- `torch`
- `torchvision`
- `xformers`

此外，对 text inpainting 路径，官方还特别要求安装修改版 diffusers：

```bash
pip install https://github.com/JingyeChen/diffusers_td2.git
```

所以最稳妥的方式是第二阶段单独建环境。

## 8.2 创建环境

```bash
conda create -n scenetap-td2 python=3.8 -y
conda activate scenetap-td2
```

## 8.3 安装 PyTorch 与 CUDA 匹配版本

这一步请根据你的 CUDA 版本去 PyTorch 官网选择安装命令。

示例：

```bash
pip install torch torchvision --index-url https://download.pytorch.org/whl/cu121
```

如果你的 CUDA 不是 `12.1`，请替换成对应版本。

## 8.4 安装 TextDiffuser-2 最小依赖

```bash
pip install accelerate==0.22.0 datasets==2.11.0 transformers==4.28.1 fschat==0.2.26
pip install diffusers==0.24.0
pip install xformers
pip install https://github.com/JingyeChen/diffusers_td2.git
```

## 8.5 安装当前仓库第二阶段需要的额外依赖

当前 `SceneTAP` 第二阶段代码还依赖：

- `openai`
- `pydantic`
- `opencv-python`
- `pillow`
- `numpy`
- `scipy`
- `tqdm`
- `pytorch-lightning`

建议一次性补齐：

```bash
pip install openai pydantic opencv-python pillow numpy scipy tqdm pytorch-lightning
```

## 8.6 关于 checkpoint

当前仓库的 `utils/text_diffuser.py` 会直接从 Hugging Face 加载这些模型：

- `JingyeChen22/textdiffuser2-full-ft-inpainting`
- `runwayml/stable-diffusion-v1-5`

因此：

- 不一定必须手动克隆 `textdiffuser-2` 仓库。
- 但必须保证网络可以访问 Hugging Face，或者你已经做好本地缓存/镜像。

## 8.7 当前 SceneTAP 实际不一定需要完整的 layout planner 训练链

这是一个很重要的细节。

`TextDiffuser-2` 官方 README 里有很多训练相关依赖，比如：

- `flash-attention`
- `FastChat`
- layout planner 训练数据

但对当前 `SceneTAP` 的主路径来说，这些并不是第一优先级必需项。

因为当前攻击流程主要使用的是：

- 区域内文字编辑 / inpainting

而不是完整重跑 `TextDiffuser-2` 的训练流程。

所以如果你的目标只是“跑通 SceneTAP 推理”，优先把：

- PyTorch
- diffusers
- transformers
- xformers
- OpenAI API

这些装对即可。

## 9. 第五步：配置 OpenAI API

当前仓库的 `utils/completion_request.py` 直接调用：

```python
from openai import OpenAI
client = OpenAI()
```

因此需要在环境中配置：

```bash
export OPENAI_API_KEY=<your_api_key>
```

如果你使用的是 PowerShell，对应写法是：

```powershell
$env:OPENAI_API_KEY="<your_api_key>"
```

如果你在 WSL 中运行，就使用前一种 `export` 写法。

## 10. 第六步：运行 SceneTAP 攻击阶段

切换到第二阶段环境：

```bash
conda activate scenetap-td2
```

然后运行：

```bash
python chatgpt_test.py \
  --model gpt-4o \
  --dataset vqav2_val2014 \
  --attack SceneTAP \
  --slider 3 \
  --filter 12 \
  --question-file <question_file_path> \
  --image-folder <image_path> \
  --log_dir logs
```

注意：

- `--slider` 和 `--filter` 需要和第一阶段生成 `som_images` 时保持一致。
- 第二阶段会按这两个参数去拼接 `som_images/...` 路径。
- 如果参数不一致，会出现找不到 SoM 输出文件的问题。

## 11. 输出结果说明

第二阶段运行后，常见输出包括：

- `logs/<model>/<dataset>/<attack>/.../images/`
  - 最终攻击图像
- `logs/<model>/<dataset>/<attack>/.../diffusion/`
  - 每个样本的若干扩散结果
- `logs/<model>/<dataset>/<attack>/.../answer.jsonl`
  - 模型回答、判定结果、攻击规划信息
- `logs/<model>/<dataset>/<attack>/.../log.txt`
  - 运行日志

## 12. 推荐的最小跑通顺序

如果你只是想先验证部署成功，建议按下面顺序：

1. 在 `scenetap-som` 环境里先跑通 `save_som_images.py`。
2. 检查 `som_images/...` 下是否同时出现图片和 `.npy` mask。
3. 在 `scenetap-td2` 环境里确认 `OPENAI_API_KEY` 已配置。
4. 再跑 `chatgpt_test.py`。

这样能把问题快速定位到：

- 是 SoM 环境问题
- 还是 TextDiffuser/OpenAI 环境问题

## 13. 常见问题

### 13.1 Windows 原生环境编译 SoM 失败

这是高概率事件。

原因是：

- `bash make.sh`
- Linux 风格 CUDA 路径
- detectron2 / 自定义扩展编译

都更偏向 Linux 环境。

建议直接改为：

- `WSL2 Ubuntu`
- 或 Linux 服务器

### 13.2 `CUDA_HOME is None`

说明 CUDA 工具链路径没有被正确识别。

处理方式：

```bash
export CUDA_HOME=/usr/local/cuda
```

然后重新编译 `SoM/ops`。

### 13.3 第二阶段找不到 `som_images`

检查以下几项是否一致：

- `dataset`
- `slider`
- `filter`
- `seed`
- `log_dir`

第二阶段会按固定路径规则读取第一阶段输出。

### 13.4 无法从 Hugging Face 下载 TextDiffuser 模型

这是网络或镜像问题，不一定是代码本身的问题。

需要确认：

- 当前环境可访问 Hugging Face
- 或者已提前把模型缓存到本地

### 13.5 OpenAI 调用失败

优先检查：

- `OPENAI_API_KEY` 是否已设置
- 当前 `openai` Python 包版本是否过旧
- 账户或网络是否能访问 OpenAI 接口

### 13.6 字体文件问题

当前仓库里 `fonts/` 目录下可见的是：

- `fonts/arial.ttf`

但 `utils/typo_attack_planner.py` 默认引用的是：

- `./fonts/arialbd.ttf`

如果运行时出现字体找不到的问题，可以考虑两种修复方式：

1. 补一个 `arialbd.ttf` 到 `fonts/` 目录。
2. 直接把代码中的字体路径改成已有的 `arial.ttf`。

## 14. 建议的环境拆分

最终建议如下。

### 环境 A：`scenetap-som`

用途：

- 编译 `SoM`
- 下载 SoM 权重
- 运行 `save_som_images.py`

关键字：

- `Python 3.10`
- `CUDA`
- `detectron2`
- `segment-anything`
- `Semantic-SAM`
- `SEEM`

### 环境 B：`scenetap-td2`

用途：

- 运行 `chatgpt_test.py`
- 调 OpenAI 生成攻击规划
- 调 `TextDiffuser` 生成对抗文字图像

关键字：

- `Python 3.8`
- `diffusers`
- `transformers`
- `xformers`
- `openai`

## 15. 一句话总结

如果只看 `SceneTAP` 本身，这两个仓库的作用可以总结为：

- `SoM`：找到“在哪个区域下手”。
- `TextDiffuser-2`：把“攻击文字自然地写进去”。

如果只看部署策略，最推荐的做法是：

- 用 `Linux/WSL2`
- 拆成 `SoM` 和 `TextDiffuser-2` 两个独立环境
- 先跑分割，再跑攻击生成

这样最稳，也最符合当前仓库的实际调用方式。
